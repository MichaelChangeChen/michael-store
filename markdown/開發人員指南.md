# 安全開發參照手冊（Codex 開發用）

> 用途：本檔案提供 Codex / AI Agent / 開發人員在產生、修改或審查程式碼時的安全開發規範。  
> 使用方式：開發新功能、改 API、改表單、改檔案上傳、改登入權限、改前後端串接時，請先參照本文件。  
> 依據文件：IS-04-21 安全開發手冊 V1.0。

---

## 0. Codex 使用規則

當 Codex 根據本專案需求產生程式碼時，請遵守以下規則：

1. 不可只做前端防護，後端必須重複驗證。
2. 不可相信任何使用者輸入，包含 URL、Query String、Route Param、Form Data、JSON Payload、Header、Cookie、檔案名稱、MIME Type。
3. 不可直接拼接 SQL、Command、HTML、URL、檔案路徑。
4. 涉及登入、權限、查詢、下載、上傳、異動、刪除時，必須考慮身份驗證、授權、Log、錯誤訊息與資料保護。
5. 前端檢核只作為 UX 輔助，真正安全檢查必須在後端執行。
6. 產生程式碼時，請優先使用框架內建安全機制，不要自行實作加密、Token、Session、驗證邏輯。
7. 若需求涉及敏感資料，預設不得放在 URL、LocalStorage、前端全域狀態或 Console Log。

# 1. 前端安全開發規範

## 1.1 權限控制失效 Broken Access Control

### 前端責任

前端可以根據使用者角色隱藏選單、按鈕、頁籤與操作入口，但這只能改善使用者體驗，不能視為真正權限控管。

### 開發規範

- 不可只靠 `v-if`、`disabled`、隱藏按鈕來保護功能。
- 不可假設使用者無法手動輸入 URL。
- 不可假設使用者無法修改 Route Param、Query String、Payload。
- 每個需要權限的 API 呼叫，都要確認後端有做權限檢查。
- 若畫面有「編輯、刪除、下載、審核、放行、退回」等功能，前端需配合角色權限顯示，但後端必須再次驗證。

### Codex 檢查重點

- 產生前端頁面時，請不要把權限判斷寫成唯一防線。
- 若產生 API 呼叫，請提醒後端需檢查該使用者是否有該資料或功能權限。

---

## 1.2 前端輸入驗證 Input Validation

### 前端責任

前端需先做基本欄位檢查，避免明顯錯誤資料送到後端。

### 常見檢核

- 必填欄位
- 最大長度 / 最小長度
- 日期格式
- 數字範圍
- Email 格式
- 電話格式
- 下拉選單值是否在允許清單內
- 檔案大小
- 檔案副檔名

### 開發規範

- 前端檢核不可取代後端檢核。
- 表單驗證失敗時，錯誤訊息要清楚，但不能暴露系統內部細節。
- 不可把後端錯誤 stack trace、SQL 錯誤、系統路徑直接顯示給使用者。

### Vue / Vuetify 注意事項

- `v-form`、`rules` 可用於 UX 驗證。
- checkbox、select、date-picker 也需檢查值是否合法。
- 表單送出前需統一驗證。
- 清除表單時要同時清除錯誤狀態。

---

## 1.3 XSS 防護

### 前端責任

避免使用者輸入內容被瀏覽器當成 HTML 或 JavaScript 執行。

### 開發規範

- 避免使用 `v-html`。
- 若一定要使用 `v-html`，必須先經過可信任的 HTML Sanitizer。
- 顯示使用者輸入資料時，預設使用文字方式渲染。
- 不可將使用者輸入直接組成 HTML 字串。
- 不可將使用者輸入直接放入 `<script>`、事件屬性、style 屬性。
- 對於 `< > ' " \ / # &` 等危險字元需留意轉義或過濾。

### Codex 檢查重點

- 不要產生直接 innerHTML / v-html 的程式碼。
- 不要把 API 回傳字串直接當 HTML 插入畫面。
- 若需求是富文字顯示，需標註必須做 Sanitization。

---

## 1.4 CSRF 防護

### 前端責任

若系統使用 Cookie 作為登入狀態，前端在送出會改變資料的請求時，需配合後端帶 CSRF Token。

### 高風險請求

- POST
- PUT
- PATCH
- DELETE
- 匯入
- 上傳
- 送審
- 放行
- 退回
- 修改密碼
- 修改權限

### 開發規範

- 不可只依賴 Cookie 自動帶出身份。
- 前端應從後端提供的位置取得 CSRF Token。
- 發送異動請求時需帶上 Token，例如 Header 或 hidden input。
- 若使用 Axios，可在 interceptor 統一帶入。

---

## 1.5 檔案上傳安全

### 前端責任

前端需先限制明顯不合法的檔案，但後端仍需做完整檢查。

### 前端需檢查

- 副檔名
- 檔案大小
- 檔案數量
- 是否必填
- UI 提示允許格式

### 開發規範

- 不可信任 `file.type`。
- 不可信任使用者提供的檔名。
- 不可只在前端限制副檔名後就認定安全。
- 上傳前可提醒使用者允許格式，例如 PDF、PNG、JPG、DOCX。

### Codex 檢查重點

- 若產生上傳元件，需加入檔案大小與副檔名限制。
- 需註明後端仍要檢查檔案頭與重新命名。

---

## 1.6 前端套件安全

### 前端責任

前端專案需定期檢查 npm 套件版本與漏洞。

### 指令

```bash
npm outdated
npm update
npm audit
npm audit fix
```

### 開發規範

- 使用 `package.json` 管理 `dependencies` 與 `devDependencies`。
- 不要安裝不明來源套件。
- 不要保留不再使用的套件。
- 更新套件後需重新測試畫面與功能。

---

## 1.7 CDN / 外部資源完整性

### 前端責任

若使用 CDN 載入 JavaScript 或 CSS，需避免外部檔案被竄改。

### 開發規範

- 優先使用本機打包資源。
- 若必須使用 CDN，建議加上 SRI。
- SRI 使用 `integrity` 屬性。
- 跨來源資源建議加 `crossorigin="anonymous"`。

### 範例

```html
<script
  src="https://example.com/example-framework.js"
  integrity="sha384-..."
  crossorigin="anonymous">
</script>
```

---

## 1.8 CSP Content Security Policy

### 前端責任

前端需配合後端設定 CSP，限制資源來源，降低 XSS 與 Clickjacking 風險。

### 開發規範

- 不要任意引入外部 script。
- 不要使用 inline script。
- 不要濫用 `unsafe-inline` 或 `unsafe-eval`。
- 圖片、字型、API、iframe 來源都需明確列入白名單。
- 若頁面需要 blob、data、YouTube、Google Maps 等來源，需明確評估。

### Codex 檢查重點

- 產生新功能時，若有外部資源，需同步提醒 CSP 要加入來源。
- 不可為了方便直接放寬為 `*`。

---

## 1.9 Clickjacking 防護

### 前端責任

前端需確認頁面是否允許被 iframe 內嵌。

### 後端 Header

- 禁止任何網站內嵌：`X-Frame-Options: DENY`
- 僅允許同源內嵌：`X-Frame-Options: SAMEORIGIN`
- CSP 寫法：`frame-ancestors 'none'` 或 `frame-ancestors 'self'`

### 開發規範

- 若頁面包含登入、個資、後台、管理功能，預設不應被外部 iframe 內嵌。
- 若有 iframe 需求，需列出允許來源。

---

## 1.10 錯誤訊息與敏感資料顯示

### 前端責任

前端顯示錯誤時，需避免暴露系統細節。

### 開發規範

- 登入失敗：顯示「帳號或密碼錯誤」，不要顯示「帳號不存在」或「密碼錯誤」。
- API 錯誤：顯示一般化訊息。
- 不顯示 SQL、Stack Trace、Server Path、Token、Session ID。
- 不在 Console Log 印出密碼、Token、個資、憑證、完整 API Response。

---

## 1.11 SSRF 相關前端注意事項

### 前端責任

若畫面允許使用者輸入 URL，需先限制格式與來源。

### 開發規範

- 不接受 `file://`、`ldap://`、`jar://` 等非預期協定。
- 不允許輸入內網 IP、localhost、127.0.0.1、metadata URL。
- 前端限制只是輔助，後端必須使用白名單檢查。

---

# 2. 後端安全開發規範

## 2.1 身份驗證 Authentication

### 後端責任

後端需確認使用者身份，並管理登入狀態。

### 可用驗證方式

- 帳號密碼
- 自然人憑證
- 行動自然人憑證
- 健保卡
- 手機 OTP
- CAPTCHA

### 開發規範

- 身份驗證資訊禁止明文傳輸。
- 登入成功後需建立安全 Session 或 Token。
- 登入失敗不可暴露帳號是否存在。
- 長期未使用帳號應停用或重新驗證。

---

## 2.2 授權 Authorization

### 後端責任

每支 API 都必須檢查使用者是否有權限操作該功能與該資料。

### 開發規範

- 不可只依賴前端隱藏按鈕。
- 每支 API 都需檢查登入狀態。
- 每支敏感 API 都需檢查角色、群組、功能權限或資料擁有權。
- 防止水平越權：使用者不可查到其他同級使用者資料。
- 防止垂直越權：一般使用者不可操作管理員功能。

### .NET 參考

- `[Authorize]`
- `[Authorize(Roles="Administrator")]`
- `[Authorize(Policy="PolicyName")]`
- `[AllowAnonymous]` 僅能用在明確公開功能，例如登入頁。

### Java 參考

- 依角色群組設定功能授權。
- 匿名角色也需明確定義可使用功能。

---

## 2.3 Session / Cookie 安全

### 後端責任

避免 Session 被竊取或濫用。

### 開發規範

- 禁止使用 URL 傳遞 Session ID。
- Cookie 設定 `HttpOnly`。
- 使用 HTTPS 時設定 `Secure`。
- 視情境設定 `SameSite=Strict` 或 `Lax`。
- Session 需有過期時間。
- 權限提升或敏感操作前可要求重新驗證。

### .NET 範例概念

```csharp
options.Cookie.HttpOnly = true;
options.Cookie.SecurePolicy = CookieSecurePolicy.Always;
options.Cookie.SameSite = SameSiteMode.Strict;
```

---

## 2.4 SQL Injection 防護

### 後端責任

避免使用者輸入被當成 SQL 指令執行。

### 開發規範

- 不可字串拼接 SQL。
- 使用參數化查詢。
- 使用 Entity Framework、PreparedStatement 或 ORM 安全查詢。
- 查詢時需限制筆數，避免資料大量外洩。
- 排序欄位、查詢欄位若由前端傳入，需使用白名單。

### 不建議

```sql
SELECT * FROM Users WHERE username = '" + username + "'
```

### 建議

- .NET：Entity Framework / SqlParameter
- Java：PreparedStatement

---

## 2.5 Form Input Validation

### 後端責任

所有表單與 API Payload 都要驗證。

### 檢查項目

- 必填
- 長度
- 格式
- 數字範圍
- 日期範圍
- 下拉選單值是否合法
- 欄位是否允許被修改

### .NET 參考

- `[Required]`
- `[StringLength]`
- `[Range]`
- `[EmailAddress]`
- `[Phone]`
- `[DataType]`
- `[Bind]` 白名單綁定，避免 Over-posting。

### Java 參考

- 前後端皆需檢核。
- 每張表單、每個欄位都要依商業邏輯檢查。

---

## 2.6 File Input Validation

### 後端責任

檔案上傳必須由後端做最終驗證。

### 檢查項目

- 檔案是否存在
- 檔案大小
- 副檔名
- MIME Type
- 檔案頭 Magic Number
- 檔名重新命名
- 儲存位置
- 權限檢查

### 常見檔案頭

| 類型 | 副檔名 | 檔案頭 |
|---|---|---|
| JPEG | jpg | FFD8FF |
| PNG | png | 89504E47 |
| GIF | gif | 47494638 |
| ZIP | zip | 504B0304 |
| PDF | pdf | 255044462D312E |

### 開發規範

- 不可直接使用原始檔名。
- 應使用隨機檔名。
- 機密檔案不可放在公開目錄。
- 上傳目錄不可允許直接執行程式。
- 下載檔案前需檢查權限。

---

## 2.7 資源保護 Resource Protection

### 後端責任

保護檔案、資料庫、API、網頁與靜態資源。

### 開發規範

- IIS 關閉目錄瀏覽。
- 使用 Request Filtering 限制上傳檔案類型與大小。
- 避免將使用者上傳檔案放在 `wwwroot`。
- Web 根目錄外的檔案需經由授權 API 下載。
- 靜態檔案若涉及權限，需透過 Middleware 或 Controller 檢查。

---

## 2.8 加密 Cryptographic Failures

### 後端責任

保護敏感資料，避免加密失效。

### 開發規範

- 先識別哪些資料是敏感資料。
- 不必要儲存的敏感資料不要儲存。
- 使用安全演算法，例如 AES。
- 不自行實作加密演算法。
- 金鑰與 IV 要安全保存。
- 不可硬編碼金鑰在程式碼中。

---

## 2.9 Hash 與密碼儲存

### 後端責任

密碼不得明碼儲存或顯示。

### 開發規範

- 密碼需經過雜湊處理。
- 實務上建議使用加鹽雜湊，例如 PBKDF2、bcrypt、Argon2。
- 不可將密碼寫入 Log。
- 不可回傳密碼給前端。

> 注意：文件提到 SHA256 可用於雜湊；但針對密碼儲存，實務上更建議使用具備 salt 與迭代成本的密碼雜湊演算法。

---

## 2.10 安全隨機數 Random Numbers

### 後端責任

Token、驗證碼、一次性密碼不可使用可預測亂數。

### 開發規範

- 不使用一般 `Random` 產生安全用途亂數。
- .NET 使用 `RandomNumberGenerator`。
- Java 使用 `SecureRandom`。

### 適用場景

- CAPTCHA
- OTP
- Password Reset Token
- Email Verify Token
- API Token
- 臨時密碼

---

## 2.11 密碼政策

### 後端責任

系統需落實密碼複雜度與帳號保護。

### 建議規範

- 密碼至少 8 碼。
- 至少包含英文與數字。
- 建議包含大小寫與特殊符號。
- 每 6 個月變更一次。
- 1 天內不得再次變更密碼。
- 不可與前一次密碼相同。
- 密碼不得明碼儲存或顯示。
- 密碼重設需重新確認身份。
- 密碼重設連結需一次性且具時效性。

---

## 2.12 登入鎖定與 CAPTCHA

### 後端責任

防止暴力破解與自動化登入攻擊。

### 開發規範

- 登入失敗達指定次數後鎖定帳號。
- Java 規範：登入失敗 5 次後，至少 15 分鐘內不可繼續嘗試。
- .NET 範例：登入失敗 3 次後，鎖定 15 分鐘。
- 登入頁應搭配 CAPTCHA。
- 鎖定、解鎖、密碼重設需記錄 Log。

---

## 2.13 安全設定缺陷 Security Misconfiguration

### 後端 / 維運責任

避免伺服器或應用程式設定不當。

### 開發規範

- 正式環境關閉 Debug Mode。
- 關閉不必要 Port。
- 移除預設帳號與預設密碼。
- 定期更新 OS、Runtime、框架、套件。
- 移除不必要功能、文件、測試頁、範例頁。
- 錯誤頁不得顯示詳細堆疊資訊。

---

## 2.14 易受攻擊和已淘汰組件 Vulnerable Components

### 後端 / 前端責任

定期檢查第三方元件與相依套件。

### .NET

- 使用 NuGet 管理套件。
- 檢查 `packages.config` 或 `.csproj` 套件版本。

### Java

- 使用 OWASP Dependency-Check。
- 檢查 Maven / Gradle 相依套件。

### 前端

- 使用 npm 管理套件。
- 使用 `npm outdated`、`npm update`、`npm audit`。

### 開發規範

- 持續監控 CVE / NVD。
- 移除不必要相依套件。
- 更新後需測試功能與相容性。

---

## 2.15 軟體及資料完整性 Software and Data Integrity

### 後端 / 前端責任

確保軟體、外部資源與資料未被竄改。

### 開發規範

- 避免使用不可信 CDN。
- 使用受信任 Repository。
- 外部 JS / CSS 建議使用 SRI。
- 系統與資料庫需定期備份。
- 需建立恢復機制。
- 需導入網站健康狀態檢查。

---

## 2.16 Log 與監控 Security Logging and Monitoring

### 後端責任

記錄重要操作與安全事件，並可供追蹤調查。

### 需記錄事件

- 登入成功 / 失敗
- 登出
- 密碼修改
- 密碼重設
- 權限異動
- 群組異動
- 查詢敏感資料
- 新增 / 修改 / 刪除
- 檔案上傳
- 檔案下載
- API 錯誤
- Exception
- 權限拒絕
- 異常存取

### 開發規範

- Log 不可記錄密碼、Token、完整個資。
- Log 需保護，避免被竄改或刪除。
- 權限拒絕與異常存取需可追蹤。
- 若系統空間可用率低於 20%，應自動警告。
- 可導入 Uptime Kuma 等監控工具。

---

## 2.17 SSRF 防護

### 後端責任

若後端會依使用者輸入的 URL 發送請求，需防止 SSRF。

### 開發規範

- 使用白名單限制可請求網域。
- 僅允許必要協定，例如 `https://`。
- 禁止 `file://`、`ldap://`、`jar://` 等非預期協定。
- 禁止請求內網 IP。
- 禁止請求 localhost、127.0.0.1。
- 禁止請求雲端 metadata URL。
- 對 URL 解析需使用安全函式庫。
- 記錄可疑請求。

---

## 2.18 Security Headers

### 後端責任

透過 HTTP Header 增加瀏覽器端防護。

### 建議 Header

```http
X-Frame-Options: SAMEORIGIN
X-Content-Type-Options: nosniff
Content-Security-Policy: frame-ancestors 'self'; default-src 'self';
Strict-Transport-Security: max-age=31536000; includeSubDomains
Referrer-Policy: strict-origin-when-cross-origin
Permissions-Policy: camera=(), microphone=(), geolocation=()
```

### 開發規範

- 不可任意放寬 CSP。
- 不可讓 CORS 開放所有來源。
- 若網站使用 HTTPS，建議啟用 HSTS。
- 若頁面不允許 iframe，使用 `DENY` 或 `frame-ancestors 'none'`。

---

## 2.19 CORS

### 後端責任

限制哪些來源可以呼叫 API。

### 開發規範

- 不建議使用 `Access-Control-Allow-Origin: *`。
- 應明確列出允許來源。
- 若允許 Cookie / Credentials，不可搭配 `*`。
- CORS 設定需依環境區分開發、測試、正式。

# 3. 前後端分工總表

| 安全項目 | 前端責任 | 後端責任 |
|---|---|---|
| 權限控制 | 控制畫面顯示 | API 必須驗證權限 |
| 輸入驗證 | UX 檢核 | 最終資料驗證 |
| XSS | 避免 v-html、避免直接渲染 HTML | 輸出編碼、CSP、資料過濾 |
| CSRF | 帶 Token | 產生與驗證 Token |
| 檔案上傳 | 檔案大小、副檔名提示 | 檔案頭、重新命名、儲存位置、權限 |
| SQL Injection | 不直接組查詢條件 | 參數化查詢、ORM、白名單 |
| Session | 不存敏感資訊於前端 | HttpOnly、Secure、SameSite、過期控制 |
| 密碼 | 不顯示、不記錄 | Hash、密碼政策、鎖定機制 |
| Log | 配合操作事件 | 寫入與保護 Log |
| 套件 | npm 檢查 | NuGet / Maven / Gradle 檢查 |
| Security Header | 避免違反 CSP 的寫法 | 設定 Header |
| CORS | 正確呼叫允許來源 | 限制允許來源 |
| SSRF | 限制 URL 輸入格式 | 白名單、封鎖內網與危險協定 |

# 4. Codex 開發前檢查清單

每次請 Codex 產生功能前，可以附上這段要求：

```text
請依據 docs/codex-security-development-guidelines.md 產生程式碼。
請特別注意：
1. 前端檢核不能取代後端檢核。
2. API 必須做身份驗證與授權。
3. 不可拼接 SQL。
4. 使用者輸入不可直接渲染 HTML。
5. 檔案上傳需檢查大小、副檔名、檔案頭，並重新命名。
6. 不可在錯誤訊息中暴露系統細節。
7. 敏感資料不可放在 URL、Console Log、LocalStorage。
8. 若有外部資源，需檢查 CSP / SRI。
9. 若有異動行為，需考慮 CSRF 與操作 Log。
10. 若有 URL 輸入，需防範 SSRF。
```
